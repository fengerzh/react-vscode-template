## 简介
本项目采用 Vitest 和 Cypress 作为核心测试框架，分别负责单元测试与端到端（E2E）测试。项目已完成从 Jest 到 Vitest 的现代化迁移，Vitest 提供了更好的性能表现和更现代化的测试体验。通过合理的配置，测试框架能够高效运行，支持别名路径解析、模块模拟、覆盖率收集等功能，为项目质量提供坚实保障。

**更新** 项目目前处于 Vitest 主导的测试配置阶段，但仍保留部分 Jest 配置文件以确保向后兼容性和渐进式迁移。

## Vitest 配置详解

Vitest 的核心配置位于 `vitest.config.ts` 文件中，定义了测试环境、转换规则、模块映射等关键选项。

### 环境配置
```ts
environment: "jsdom"
```
使用 jsdom 模拟浏览器环境，适合 React 组件测试，提供 DOM API 和浏览器原生对象。

### 设置文件配置
```ts
setupFiles: ["./jest.setup.ts"]
```
指定在每个测试环境初始化后执行的脚本文件，继承了原有的 `jest.setup.ts` 初始化逻辑，确保测试环境的一致性。

### 测试文件匹配
```ts
include: ["src/**/*.{test,spec}.{ts,tsx}"]
```
指定测试文件的匹配模式，Vitest 支持 `.test.*` 和 `.spec.*` 扩展名，涵盖 TypeScript 和 TSX 文件。

### 排除配置
```ts
exclude: ["cypress/**/*"]
```
排除 Cypress 目录下的文件，避免 Vitest 执行 E2E 测试。

### 覆盖率配置
```ts
coverage: {
  provider: "v8",
  reporter: ["text", "lcov", "clover"],
  include: ["src/**/*.{ts,tsx}"],
  exclude: [
    "src/**/*.mock.ts",
    "src/**/*.test.ts",
    "src/**/*.spec.ts",
    "src/types/**/*",
  ],
}
```
使用 V8 引擎提供覆盖率数据，支持多种报告格式，包括文本、LCOV 和 Clover 格式，确保代码覆盖率的准确统计。

### 别名配置
```ts
resolve: {
  alias: {
    "@": path.resolve(__dirname, "src"),
  },
}
```
实现 `@/` 别名支持，允许开发者使用 `@/components/Home` 而非相对路径，简化导入语句。

## Cypress 配置详解

Cypress 配置文件 `cypress.config.ts` 定义了 E2E 和组件测试的行为。

### baseUrl 配置
```ts
baseUrl: "http://localhost:3123"
```
设置应用的基础 URL，所有 `cy.visit()` 调用将基于此地址。例如，`cy.visit("/login")` 实际访问 `http://localhost:3123/login`，便于在不同环境中切换测试目标。

### specPattern 配置
```ts
specPattern: "cypress/e2e/**/*.{js,jsx,ts,tsx}"
```
指定测试文件的匹配模式，Cypress 将自动发现并运行 `cypress/e2e` 目录下所有符合条件的测试脚本，支持 JavaScript 和 TypeScript。

### setupNodeEvents 配置
```ts
setupNodeEvents(on, config) {
  return pluginConfig(on, config);
}
```
该钩子允许扩展 Cypress 的 Node.js 环境功能，如修改配置、注册任务、监听事件等。当前项目通过 `plugins/index.ts` 提供插件逻辑，保持配置简洁。

### component 测试配置
```ts
component: {
  devServer: {
    framework: "react",
    bundler: "vite",
  },
}
```
启用组件测试模式，使用 Vite 作为打包器，支持在隔离环境中快速测试单个 React 组件，提升开发效率。

#### ResizeObserver Polyfill
```ts
class ResizeObserverPolyfill {
  observe() {}
  unobserve() {}
  disconnect() {}
}
window.ResizeObserver = ResizeObserverPolyfill;
```
为现代组件提供 ResizeObserver API 的 polyfill，确保依赖该 API 的组件正常工作。

#### 控制台静音机制
通过重写 `console.error` 和 `console.log`，过滤掉已知的无关警告，减少测试输出噪声，聚焦关键错误：

```ts
const originalError: typeof console.error = console.error;
const shouldSilenceConsole = (args: unknown[]): boolean => {
  const text = args
    .map((a) => {
      // 更稳健地识别 Error-like 对象（跨 realm 不用 instanceof）
      if (
        a
        && typeof a === "object"
        && (
          Object.prototype.hasOwnProperty.call(a, "message")
          || Object.prototype.hasOwnProperty.call(a, "stack")
        )
      ) {
        const anyA = a as { message?: unknown; stack?: unknown };
        const parts = [anyA.message, anyA.stack]
          .filter((p) => typeof p === "string")
          .join(" ");
        if (parts) return parts as string;
      }
      if (typeof a === "string") return a;
      try {
        return JSON.stringify(a);
      } catch {
        return String(a);
      }
    })
    .join(" ");

  return [
    "ReactDOMTestUtils.act",
    "Not implemented: navigation",
    "API请求错误",
    "window.matchMedia is not a function",
    "window.getComputedStyle",
    // 预期内的接口失败场景（如 401）产生的 AxiosError 输出
    "AxiosError",
    "Request failed with status code 401",
  ].some((needle) => text.includes(needle));
};
console.error = (
  ...args: Parameters<typeof console.error>
) => {
  if (shouldSilenceConsole(args)) return;
  originalError(...args);
};
```

#### 全局 polyfill
为 Node.js 环境补充 `TextEncoder` 和 `TextDecoder`，避免因缺失 Web API 导致测试失败：

```ts
const g = global as unknown as {
  TextEncoder: typeof TextEncoder;
  TextDecoder: typeof TextDecoder;
};
g.TextEncoder = TextEncoder;
g.TextDecoder = TextDecoder;
```

#### 全局对象 Mock
- `window.matchMedia`：模拟响应式媒体查询，防止组件因缺少该 API 而报错
- `window.getComputedStyle`：返回空样式对象，避免布局相关错误
- `window.scrollTo`：mock 滚动行为，避免副作用

这些设置确保所有测试在一致、可控的环境中运行。

## localStorage Mock 实现详解

### 核心实现机制
在 `jest.setup.ts` 中，项目实现了完整的 localStorage mock 机制，专门用于支持 Zustand 持久化存储：

```ts
// Setup localStorage mock for zustand persist
const localStorageStore: Record<string, string> = {};
const mockLocalStorage = {
  getItem: (key: string) => localStorageStore[key] ?? null,
  setItem: (key: string, value: string) => { localStorageStore[key] = String(value); },
  removeItem: (key: string) => { delete localStorageStore[key]; },
  clear: () => { Object.keys(localStorageStore).forEach(k => delete localStorageStore[k]); },
  length: 0,
  key: (_index: number) => null,
};
vi.stubGlobal("localStorage", mockLocalStorage);
```

### 实现特点
1. **内存存储**：使用内存对象 `localStorageStore` 存储键值对，避免持久化到磁盘
2. **类型安全**：所有值都转换为字符串存储，确保与真实 localStorage 的行为一致
3. **完整 API 支持**：实现 `getItem`、`setItem`、`removeItem`、`clear`、`length`、`key` 等完整方法
4. **全局可用**：通过 `vi.stubGlobal` 将 mock 注入到全局作用域

### 使用场景
这个 mock 主要用于测试 Zustand 状态管理的持久化功能，确保状态可以在测试环境中正确保存和恢复。

### 细粒度控制替代方案
对于需要更精确控制的测试场景，项目还提供了 `matchMedia.mock.ts` 文件：

```ts
import { vi } from "vitest";

Object.defineProperty(window, "matchMedia", {
  writable: true,
  value: vi.fn().mockImplementation((query: string) => ({
    matches: false,
    media: query,
    onchange: null,
    addListener: vi.fn(),
    removeListener: vi.fn(),
    addEventListener: vi.fn(),
    removeEventListener: vi.fn(),
    dispatchEvent: vi.fn(),
  })),
});
```

这种细粒度的 mock 方案允许开发者只在需要的测试文件中引入特定的环境模拟，实现更精确的测试控制。

## 测试配置最佳实践

### 测试环境变量配置
建议通过 `.env.test` 文件管理测试专用环境变量，并在 `vitest.config.ts` 或 `cypress.config.ts` 中读取，避免硬编码敏感信息。

### 模块模拟策略
- **第三方库**：使用 `vi.mock()` 模拟外部依赖（如 `axios`、`antd`），确保测试不依赖网络或 UI 库实现
- **API 请求**：结合 `axios-mock-adapter` 模拟 HTTP 响应，验证不同状态码下的应用行为
- **React 组件**：使用 `@testing-library/react` 的 `render` 函数和 `fireEvent` 工具进行交互测试

### 代码覆盖率集成
Vitest 内置覆盖率工具，通过以下配置优化：
```ts
coverage: {
  provider: "v8",
  reporter: ["text", "lcov", "clover"],
  include: ["src/**/*.{ts,tsx}"],
  exclude: [
    "src/**/*.mock.ts",
    "src/**/*.test.ts",
    "src/**/*.spec.ts",
    "src/types/**/*",
  ],
}
```
使用 V8 引擎提供覆盖率数据，支持多种报告格式，确保新增代码符合质量标准。

### 并行测试执行
利用 Vitest 的并行执行能力，结合 `--maxConcurrency` 参数提升大型测试套件的运行速度。

### 性能优化
- 使用 `vi.spyOn()` 替代 `jest.spyOn()` 进行方法监控
- 使用 `vi.fn()` 替代 `jest.fn()` 创建模拟函数
- 使用 `vi.mock()` 替代 `jest.mock()` 进行模块模拟

### 环境隔离策略
- **全局初始化**：通过 `jest.setup.ts` 提供统一的全局环境配置
- **细粒度控制**：通过 `matchMedia.mock.ts` 等文件实现特定场景的环境模拟
- **测试专用配置**：为不同类型的测试（单元测试、组件测试、E2E测试）提供专门的配置
